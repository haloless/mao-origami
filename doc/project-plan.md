# 项目规划设计

## 1. 项目目标

项目目标拆分为两个阶段能力：

### 阶段一：折纸演示器

- 内置折纸模型
- 按步骤播放折叠动画
- 支持上一步、下一步、单步重播、自动播放
- 支持旋转、缩放和查看当前状态
- 通过高亮、文字和语音帮助儿童理解步骤

### 阶段二：用户折纸编辑器 + 演示器

- 用户创建或修改折纸文档
- 用户编辑步骤、折线、说明文字、镜头和动画
- 用户保存草稿、回放预览、导出成可分享作品

阶段一必须为阶段二预留结构，不能把播放器做成一次性页面逻辑。

## 2. 产品边界

当前路线明确不做以下内容：

- 不做 OCR 识别互联网步骤图
- 不做通用图像推理
- 不做高保真连续物理模拟

当前路线重点做以下内容：

- 确定性的步骤驱动折纸动画
- 面向儿童教学的交互回放体验
- 后续可扩展的用户编辑能力

## 3. 核心设计原则

### 3.1 播放器和编辑器共享同一核心模型

播放器与编辑器都应基于 `OrigamiDocument` 工作，而不是各自维护一套数据结构。

### 3.2 核心逻辑与渲染层分离

折纸命令解释、状态计算、播放控制应在纯 TypeScript 模块中实现。
Three.js 或 React 只负责展示结果。

### 3.3 先做稳定播放，再做编辑体验

只有当步骤模型、状态缓存和播放器行为足够稳定，编辑器才有可靠基础。

### 3.4 从一开始预留编辑器所需能力

即使第一阶段 UI 只有播放器，也必须在架构层预留：

- 选择状态
- 命令历史
- 文档校验
- 草稿保存
- 结构化导入导出

## 4. 推荐技术栈

考虑当前目标是 Web 演示器优先、后续扩展编辑器，建议使用前端主导方案。

### 4.1 前端

- `Vite`
- `TypeScript`
- `React`
- `@react-three/fiber`
- `@react-three/drei`
- `Zustand`
- `Zod`

说明：

- `React` 适合承载步骤面板、时间轴、属性面板、编辑器侧栏。
- `react-three-fiber` 适合把 3D 视图嵌入整个应用状态体系。
- `Zod` 可用于文档校验，后续对编辑器非常重要。

### 4.2 备选依赖

- `rabbit-ear`：可作为折纸几何能力参考或局部底层工具。
- `valtio` 或 `jotai`：若后续状态拆分需要，也可替代部分局部状态方案。
- `vitest`：单元测试。
- `playwright`：播放器和编辑器关键流程测试。

### 4.3 后端

第一阶段建议不强依赖后端。

后续仅在以下需求出现时再加入后端：

- 用户账户
- 云端存储
- 作品分享
- 协作编辑
- 模型资源发布

如果后续加入后端，建议使用：

- `Node.js + Fastify` 或 `Python + FastAPI`

但当前不应让后端成为首阶段阻碍。

## 5. 建议的项目结构

为了给编辑器预留位置，建议一开始就按“核心库 + 应用层”组织，而不是把所有逻辑都堆进页面组件。

```text
mao-origami/
  doc/
  public/
  src/
    app/
      routes/
      providers/
      store/
    features/
      player/
      viewer3d/
      step-panel/
      playback-controls/
      annotations/
      editor-shell/
      editor-timeline/
      editor-inspector/
      editor-selection/
    core/
      document/
      geometry/
      commands/
      playback/
      validation/
      serialization/
    assets/
      models/
      audio/
      textures/
    styles/
    main.tsx
```

### 5.1 core 层职责

`core` 是项目长期最重要的部分，应保持与 UI 框架低耦合。

- `document/`：文档类型、schema、导入导出
- `geometry/`：面片、折线、状态与几何计算
- `commands/`：每一步命令解释和状态转移
- `playback/`：播放控制器、时间轴、缓存跳步
- `validation/`：文档和步骤校验
- `serialization/`：JSON/FOLD 兼容层

### 5.2 features 层职责

- `player/`：播放器页面协调逻辑
- `viewer3d/`：3D 模型渲染、镜头、交互
- `step-panel/`：步骤列表与当前步骤说明
- `playback-controls/`：播放控制条
- `annotations/`：文字、箭头、高亮覆盖层
- `editor-shell/`：未来编辑器容器
- `editor-timeline/`：未来步骤编排面板
- `editor-inspector/`：未来属性编辑面板
- `editor-selection/`：未来选区与命中逻辑

阶段一可以只实现前五项，但后四项目录应预留。

## 6. 核心模块设计

### 6.1 文档模型模块

职责：

- 定义 `OrigamiDocument`
- 加载和保存 JSON 文档
- 做 schema 版本兼容
- 为编辑器提供默认空文档和补全逻辑

### 6.2 命令执行模块

职责：

- 接收 `StepCommand`
- 根据当前状态生成下一状态
- 生成动画所需的插值参数
- 为校验模块暴露可执行性检查

这里必须做成纯函数接口，避免把播放状态和文档状态耦在一起。

### 6.3 播放控制模块

职责：

- 当前步骤索引管理
- 单步播放与回放
- 自动播放
- 跳转到指定步骤
- 通过状态缓存快速恢复结果态

建议核心接口形态：

```ts
type PlaybackController = {
  play(): void;
  pause(): void;
  next(): void;
  previous(): void;
  replayStep(stepId?: string): void;
  seekStep(stepIndex: number): void;
  setSpeed(speed: number): void;
};
```

### 6.4 3D 视图模块

职责：

- 渲染面片和折线
- 应用 `OrigamiState` 中的变换
- 实现旋转、缩放、平移
- 支持高亮目标折线和面片
- 提供编辑器命中检测基础能力

这一层要避免直接理解业务语义，只消费标准状态。

### 6.5 编辑器预留模块

即便第一阶段不实现完整 UI，也建议从架构上保留以下边界：

- `SelectionService`：当前选中的面、折线、点、步骤
- `CommandHistoryService`：撤销与重做
- `DocumentDraftService`：草稿变更、脏状态、自动保存
- `ValidationService`：实时校验和错误定位
- `ImportExportService`：导入导出 JSON，后续兼容 FOLD

这些服务先定义接口，第二阶段逐步补实现。

## 7. 播放器页面规划

第一阶段建议的页面布局：

- 左侧：3D 折纸视图
- 右侧：步骤列表和当前说明
- 底部：播放控制栏
- 顶部：模型信息、速度、重播按钮

建议首版功能：

- 上一步
- 下一步
- 重播当前步
- 自动播放/暂停
- 旋转缩放
- 当前折线高亮
- 文字提示
- 可选语音朗读

## 8. 编辑器预留规划

这是本项目最重要的非功能性要求之一。

为了确保未来支持“用户折纸编辑器 + 演示器”，阶段一需要明确保留下列扩展位：

### 8.1 编辑器入口

应用路由建议从第一版开始就预留：

- `/play/:modelId`
- `/edit/:modelId`

即使 `/edit/:modelId` 暂时只显示占位页面，也应在路由和状态层留好入口。

### 8.2 时间轴模型

播放器当前步骤列表不应只是一组 UI 文本，而应基于真实 `steps` 数组渲染。这样未来编辑器才能直接对步骤做：

- 插入
- 删除
- 重排
- 复制
- 分组

### 8.3 选择与命中系统

3D 视图未来会承担编辑器中的点击选中，因此第一阶段就应给面片、折线、点分配稳定 ID，并确保 3D 实体能反向映射回文档对象。

### 8.4 属性面板数据形态

步骤说明、动画时长、镜头预设、箭头提示等字段应统一进入 `presentation` 和 `editorMeta`，避免将来从零迁移散落在 React 组件里的状态。

### 8.5 草稿能力

建议从第一版开始就区分：

- 内置官方模型
- 用户草稿模型

即使本地只用 `localStorage` 或文件下载保存，也应让文档具备 `draft/published` 概念。

## 9. 分阶段实施路线

### Milestone 1：数据模型与静态播放器

- 落地文档 schema
- 完成 1 到 2 个示例模型
- 渲染静态步骤结果
- 支持步骤切换

### Milestone 2：动画播放器

- 实现 `fold`、`unfold`、`flip_model`、`rotate_model`
- 实现单步重播和自动播放
- 支持速度控制与状态缓存

### Milestone 3：教学体验增强

- 加入折线高亮
- 加入箭头和说明气泡
- 加入音频和儿童友好界面细节

### Milestone 4：编辑器基础设施

- 引入 `/edit/:modelId` 页面
- 打通文档读取、草稿保存、基本校验
- 支持步骤重命名、说明编辑、时长调整

### Milestone 5：编辑器核心功能

- 选择面片和折线
- 新增或修改步骤命令
- 撤销重做
- 实时预览与发布

## 10. 风险与控制策略

### 10.1 风险：过早追求物理模拟

控制策略：

- 第一阶段只做确定性步骤动画
- 使用面片变换与层级规则，不做连续纸张力学

### 10.2 风险：播放器和编辑器分叉

控制策略：

- 强制共享 `OrigamiDocument`
- 核心逻辑下沉到 `core`
- UI 组件只消费标准状态

### 10.3 风险：步骤命令表达力不足

控制策略：

- MVP 先支持少量基础命令
- 通过联合类型持续扩展高级折法
- 为命令保留扩展字段和校验层

### 10.4 风险：后续编辑器接入代价过高

控制策略：

- 稳定 ID
- 命令历史
- 文档草稿态
- 选择与命中系统前置设计

## 11. 当前建议结论

本项目最稳妥的路线是：

- 第一阶段做前端主导的折纸演示器。
- 从第一天起采用统一的 `OrigamiDocument` 数据模型。
- 把命令执行、播放控制、校验和序列化作为 `core` 能力沉淀。
- 在目录结构、路由、状态模型和服务接口上明确为用户折纸编辑器预留位置。

这条路线的好处是：第一阶段能尽快做出儿童可用的演示产品；第二阶段在不推翻已有系统的前提下，自然演进为“用户折纸编辑器 + 演示器”。
